# CT-52 Test Notes

## Test files created

### 1. `src/test/theme/board_sizing_helpers_test.dart`

Unit tests for the three pure-function helpers added to `spacing.dart`. Tests the mathematical contracts without any widget involvement.

**What it covers:**

- **Constants sanity checks:** `kBoardHorizontalInset` is 4.0, `kMaxBoardSize` is 600.0, `kBoardHorizontalInsets` has correct symmetric horizontal padding.
- **`boardSizeForWidth`:**
  - Typical phone widths (390, 375, 320) produce `width - 2 * 4`.
  - Wide viewports (700, 1920) clamp to `kMaxBoardSize` (600).
  - Boundary at exactly `kMaxBoardSize + 2 * inset` (608px).
  - Zero and very small widths (less than `2 * inset`) clamp to 0.
- **`boardSizeForConstraints`:**
  - Default `widthFraction` of 0.5.
  - Custom `widthFraction` of 0.6.
  - Height-constrained scenarios (height < width * fraction).
  - Width-constrained scenarios (width * fraction < height).
  - Ultrawide clamping to `kMaxBoardSize`.
  - Zero constraints return 0.
  - Determinism (same inputs produce same outputs).
- **`boardSizeForNarrow`:**
  - Phone screen with default `maxHeightFraction` (1.0).
  - Browser-style `maxHeightFraction` (0.4) on typical and short phones.
  - Height fraction constraining more than width.
  - Width constraint dominating over height.
  - Wide viewport clamping.
  - Zero width and zero height edge cases.
- **Cross-function consistency:**
  - `boardSizeForWidth` and `boardSizeForNarrow` produce the same result on a tall screen with default `maxHeightFraction`.
  - All helpers never exceed `kMaxBoardSize`.
  - All helpers never return negative values.

### 2. `src/test/layout/board_layout_wide_test.dart`

Widget test that pumps all four board screens at a wide viewport (`1024x768`) and asserts board-size consistency.

**What it covers:**

- All four screens (AddLine, Drill, FreePractice, RepertoireManager) produce the same board size at a wide viewport -- mirrors the existing narrow test pattern.
- The board is square at wide viewport.
- The board does not exceed `kMaxBoardSize` at wide viewport.
- Separate ultrawide test (`1920x1080`) verifies the `kMaxBoardSize` cap prevents absurdly large boards on ultrawide monitors.

### 3. `src/test/layout/board_layout_narrow_sizing_test.dart`

Widget test that verifies the narrow-layout board size matches the `boardSizeForWidth` helper output, proving screens actually use the shared helper.

**What it covers:**

- Board width equals `boardSizeForWidth(screenWidth)` at three phone sizes: typical (390x844), small/SE (375x667), narrow (320x568).
- Board is square at all tested phone sizes.
- Board is nearly edge-to-edge: the gap between board and screen width equals exactly `2 * kBoardHorizontalInset` (8dp total).

## Acceptance criteria that could not be tested automatically

1. **Spec file updates** (`architecture/board-layout-consistency.md`, `design/ui-guidelines.md`): These are prose documents. Their content can be reviewed in code review but cannot be meaningfully validated by automated tests.

2. **Visual appearance ("appears nearly edge-to-edge")**: The widget tests verify pixel dimensions match the formula, which is a strong proxy. However, true visual verification (e.g., that padding looks correct, that the board does not overlap other elements, that the layout is aesthetically pleasing) requires manual inspection or screenshot testing, which is out of scope for these unit/widget tests.

3. **Line-label area, move pills, and action buttons layout correctness**: The plan's Step 7 calls for manual verification that these elements still lay out correctly with the larger board. This is a visual/functional concern that would require more elaborate widget tests with specific layout assertions, which are outside CT-52's scope.
