# CT-63: Test Notes

## Test files modified

- `src/test/screens/home_screen_test.dart` — Appended four new test groups (Steps 7-10) and added import for `RepertoireCard`.

## What each test covers

### Group: "HomeScreen - rename repertoire dialog" (Step 7, 7 tests)

1. **Rename dialog opens from context menu** — Opens popup menu on a RepertoireCard, taps "Rename", verifies the dialog appears with the title "Rename repertoire" and the text field pre-filled with the current name.
2. **Rename validates empty name** — Clears the text field in the rename dialog, verifies the "Rename" confirm button is disabled (onPressed is null).
3. **Rename confirms and updates list** — Enters a new name "Bravo", taps "Rename", verifies the list shows "Bravo" and the old name "Alpha" is gone.
4. **Rename cancel does not modify** — Enters a different name but taps "Cancel", verifies the original name is still displayed.
5. **Rename duplicate warning** — With two repertoires ("Alpha", "Beta"), opens rename on "Alpha", enters "Beta", verifies the error text "A repertoire with this name already exists" appears but the "Rename" button remains enabled (soft warning).
6. **Rename duplicate warning case-insensitive** — Same setup, enters "beta" (lowercase), verifies the same warning appears.
7. **Rename duplicate warning excludes current name** — Opens rename on "Alpha", re-enters "Alpha", verifies no warning text is shown.

### Group: "HomeScreen - delete repertoire dialog" (Step 8, 4 tests)

1. **Delete dialog opens from context menu** — Opens popup menu, taps "Delete", verifies the confirmation dialog with title "Delete repertoire" and content referencing the repertoire name.
2. **Delete confirms and removes** — Confirms deletion, verifies the repertoire is removed and the screen transitions to the empty state.
3. **Delete cancel does not remove** — Taps "Cancel" on the confirmation dialog, verifies the repertoire is still displayed.
4. **Delete targets correct repertoire in multi-card list** — With two repertoires, opens popup on the second card ("Beta"), deletes it, verifies "Alpha" remains and "Beta" is gone.

### Group: "HomeScreen - create additional repertoire" (Step 9, 3 tests)

1. **FAB appears when repertoires exist** — Pumps with one repertoire, verifies a FloatingActionButton with tooltip "Create repertoire" exists.
2. **Create dialog opens and creates second repertoire** — Taps the FAB, enters a name, confirms, verifies two RepertoireCard widgets are now shown and the screen stays on HomeScreen (does not navigate to browser).
3. **Duplicate name shows warning** — Taps FAB, enters an existing name, verifies the error text appears but the "Create" button remains enabled.

### Group: "HomeScreen - multi-repertoire list" (Step 10, 5 tests)

1. **Renders correct number of RepertoireCard widgets** — Pumps with three repertoires, verifies `find.byType(RepertoireCard)` finds three.
2. **Each card shows its own name and due count** — Pumps with two repertoires with different due counts, verifies each card's name and badge text ("1 due", "2 due").
3. **Rename targets correct repertoire (non-first card)** — Opens rename on the second card ("Beta"), renames to "Gamma", verifies "Alpha" is unchanged and "Gamma" appears.
4. **Navigation uses correct repertoire ID (non-first card)** — Taps the name on the second card, verifies navigation to RepertoireBrowserScreen (uses real DB to validate the navigation path works).
5. **Start Drill on non-first card uses correct repertoire ID** — With only the second repertoire having due cards, taps "Start Drill" on the second card, verifies DrillScreen launches.
