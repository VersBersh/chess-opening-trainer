# CT-60: Test Notes

## Test files modified

- `src/test/screens/add_line_screen_test.dart`

## What each test covers

### Updated: `"Add name" does not persist and stays on screen`

Added assertion that `InlineLabelEditor` is visible (`findsOneWidget`) after tapping "Add name" in the no-name warning dialog. This covers the single-pill (leaf-focused) auto-open case. Existing assertions (dialog dismissed, no moves persisted, screen still visible) are preserved.

### New: `"Add name" opens editor when a non-leaf pill is focused`

Covers the case where the user plays two moves (e4, e5), taps the first pill (e4) to move focus away from the leaf, then taps Confirm and chooses "Add name". Asserts:
- The no-name dialog appears and is dismissed after tapping "Add name".
- `InlineLabelEditor` is visible after dismissal.
- No moves are persisted to the database.

This tests a distinct scenario from the single-pill test: focus is on a non-leaf pill, verifying the editor opens on whichever pill is currently focused.

### Fixed: `"Add name" short-circuits before parity validation`

Replaced the false-positive assertion `expect(find.text('Line parity mismatch'), findsNothing)` (which matched a string never rendered in the UI) with assertions against the actual parity warning text:
- `expect(find.textContaining('should end on a'), findsNothing)`
- `expect(find.textContaining('Flip and confirm as'), findsNothing)`
