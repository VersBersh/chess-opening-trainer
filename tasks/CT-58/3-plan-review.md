**Issues**
1. Major — **Step 3:** the direct-child colour rule is wrong at the root position. Root moves come from `getRootMoves()`, so their `parentMoveId` is `null`; using `child.parentMoveId == currentMoveId` will classify initial-position repertoire moves as transposition arrows instead of direct children. Evidence: [src/lib/models/repertoire.dart](C:/code/draftable/chess-3/src/lib/models/repertoire.dart#L122) and the existing branch-arrow logic in [src/lib/controllers/repertoire_browser_controller.dart](C:/code/draftable/chess-3/src/lib/controllers/repertoire_browser_controller.dart#L267). Fix: decide “direct vs transposition” from membership in the computed direct-child set / ID set, not from comparing parent IDs.

2. Major — **Step 6, test 4:** the proposed transposition fixture does not actually exercise transposition child arrows. `getChildrenAtPosition()` returns the children of nodes at that position, not the nodes that reach that position themselves ([src/lib/models/repertoire.dart](C:/code/draftable/chess-3/src/lib/models/repertoire.dart#L109)). With `['e4', 'd6', 'd4']` and `['d4', 'd6', 'e4']`, both lines stop at the shared position, so there are no children to render as lighter arrows. Fix: seed at least one child under a transposed node and assert arrows from that shared parent position.

3. Minor — **Step 3:** the deduplication key is inconsistent with the intended move identity. The plan says dedupe by from/to/promotion, but the sample key only uses from/to, which would merge distinct promotion moves on the same squares. Related move matching elsewhere already treats promotion as part of identity: [src/lib/controllers/repertoire_browser_controller.dart](C:/code/draftable/chess-3/src/lib/controllers/repertoire_browser_controller.dart#L382). Fix: include `promotion?.name` in the key, and specify that a darker direct-child arrow replaces an existing lighter transposition arrow for the same key.
