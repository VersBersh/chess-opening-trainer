## Acceptance Criteria That Could Not Be Tested Automatically

| Criteria | Reason |
|----------|--------|
| "User can see what they are typing in the filter" | This is a visual/UX assertion. The tests verify the `TextField` widget is present and not occluded by board collapse, but cannot assert that rendered pixels are literally visible to a human. Widget test `findsOneWidget` confirms the widget is in the tree and hit-testable. |
| "User can see and select from the dropdown suggestions" | The dropdown overlay positioning depends on `_computeDropdownLayout` which uses `RenderBox.localToGlobal` â€” hard to verify pixel positions in a widget test. The existing filter interaction tests cover dropdown selection functionality; visual positioning requires manual device testing. |
| Smooth animated transition (AnimatedSize over 200ms) | Widget tests can verify the before/after states (collapsed vs expanded) but cannot meaningfully assert animation smoothness. The `AnimatedSize` widget and its `duration`/`curve` parameters would need a visual inspection or golden test. |
| `resizeToAvoidBottomInset` conditional flag (Step 5) | The plan leaves this as a conditional decision ("verify during development"). Tests assert the intended layout outcomes regardless of whether this flag is set â€” if the board collapses correctly and the filter is visible, the test passes.
