- **Verdict** — `Needs Fixes`
- **Issues**
1. **Major — Hidden coupling / correctness:** `performReroute()` derives the target node from the focused pill for conflict checking, but derives the persisted reroute target from `engine.lastExistingMoveId`. Those are not the same thing when the user focuses an earlier saved pill. In that case the pre-check runs against one node, while the repository re-parents children under a different node. See [add_line_controller.dart](/C:/code/draftable/chess-4/src/lib/controllers/add_line_controller.dart#L803), [add_line_controller.dart](/C:/code/draftable/chess-4/src/lib/controllers/add_line_controller.dart#L818), [add_line_controller.dart](/C:/code/draftable/chess-4/src/lib/controllers/add_line_controller.dart#L835), [line_persistence_service.dart](/C:/code/draftable/chess-4/src/lib/services/line_persistence_service.dart#L190), and [local_repertoire_repository.dart](/C:/code/draftable/chess-4/src/lib/repositories/local/local_repertoire_repository.dart#L421). This is a semantic coupling bug: reroute only works correctly if the focused pill happens to equal the last existing move. Fix by computing an explicit `newConvergenceId` from the focused pill and passing that through the service/repository API, instead of overloading `anchorMoveId` for both “parent of first inserted move” and “final convergence node when nothing is inserted.” Add a controller/widget test for rerouting from an earlier saved focus point, not just the last pill.
2. **Minor — DRY / embedded design:** `getRerouteInfo()` reconstructs path text by manually walking `existingPath`, `followedMoves`, and `bufferedMoves`, and re-implements move-notation formatting for buffered moves. That duplicates path/notation knowledge already living in `RepertoireTreeCache` and `LineEntryEngine`, and it makes the controller understand low-level engine internals that should stay encapsulated. See [add_line_controller.dart](/C:/code/draftable/chess-4/src/lib/controllers/add_line_controller.dart#L739) and [repertoire.dart](/C:/code/draftable/chess-4/src/lib/models/repertoire.dart#L157). Fix by extracting a single helper that produces an active-path description up to a pill index; the controller should ask for that, not assemble it itself.
3. **Minor — File size / SRP:** this change adds more responsibilities to files that are already well past the 300-line smell threshold: [add_line_controller.dart](/C:/code/draftable/chess-4/src/lib/controllers/add_line_controller.dart), [add_line_screen.dart](/C:/code/draftable/chess-4/src/lib/screens/add_line_screen.dart), and [local_repertoire_repository.dart](/C:/code/draftable/chess-4/src/lib/repositories/local/local_repertoire_repository.dart). The reroute flow is reasonable, but continuing to pack orchestration, result types, UI flow, and persistence details into these files makes the architecture harder to read from code alone. Fix by extracting reroute-specific collaborators: for example a small reroute coordinator/service for controller logic, and a dedicated widget/helper for transposition-warning actions.
