**Verdict** — `Needs Revision`

**Issues**
1. **Critical — Step 5 / Step 7 / Step 10:** the plan exposes reroute for any same-opening transposition, but the proposed reroute algorithm only moves **children** of the matched node. `TranspositionMatch.moveId` is the move that already reaches the transposed position, not a continuation node ([line_entry_engine.dart](/C:/code/draftable/chess-4/src/lib/services/line_entry_engine.dart#L64)), and the current UI shows `Reroute` for every same-opening match ([add_line_screen.dart](/C:/code/draftable/chess-4/src/lib/screens/add_line_screen.dart#L724), [add_line_screen_test.dart](/C:/code/draftable/chess-4/src/test/screens/add_line_screen_test.dart#L2437)). For a leaf match, `rerouteLine` would have nothing to re-parent, so the action becomes a no-op or produces the wrong semantics. Fix: either gate reroute to matches where the matched node has children (`!treeCache.isLeaf(match.moveId)`) and update the existing widget/spec tests accordingly, or design a separate leaf-reroute flow that reparents the node itself and handles review-card consequences.

2. **Major — Step 3:** the controller step is still wrong about which buffered moves belong to the reroute target. The plan text says to use `engine.bufferedMoves`, but the controller supports focusing an earlier pill while later pills remain visible ([add_line_controller.dart](/C:/code/draftable/chess-4/src/lib/controllers/add_line_controller.dart#L493), [add_line_controller.dart](/C:/code/draftable/chess-4/src/lib/controllers/add_line_controller.dart#L815)). In that state, reroute must persist only the buffered prefix up to `focusedPillIndex`; using the whole buffer would create the wrong convergence node and can discard unrelated post-focus work. The risk note notices this, but the implementation step does not. Fix: make the focused-pill slice part of the concrete step, and define the behavior when buffered moves exist after the convergence point before any repository/service work is implemented.

3. **Major — Step 1:** changing `RepertoireRepository` means updating more than the production repository and the one fake mentioned in context. There are additional test doubles implementing the interface in [pgn_importer_test.dart](/C:/code/draftable/chess-4/src/test/services/pgn_importer_test.dart#L100), [drill_screen_test.dart](/C:/code/draftable/chess-4/src/test/screens/drill_screen_test.dart#L77), [home_screen_test.dart](/C:/code/draftable/chess-4/src/test/screens/home_screen_test.dart#L25), [drill_filter_test.dart](/C:/code/draftable/chess-4/src/test/screens/drill_filter_test.dart#L80), and [deletion_service_test.dart](/C:/code/draftable/chess-4/src/test/services/deletion_service_test.dart#L12). Without a plan step for those, analysis/tests will fail as soon as the interface changes. Fix: add an explicit follow-up step to update all `implements RepertoireRepository` fakes/spies or introduce a shared base fake.

4. **Minor — Step 4 / Step 6:** the dialog copy is inconsistent with the available API. The plan says the confirmation dialog should say “Move N continuation move(s)”, but the proposed implementation uses `countDescendantLeaves()` ([repertoire.dart](/C:/code/draftable/chess-4/src/lib/models/repertoire.dart#L210)), which counts **leaf lines**, not moved nodes. Existing UI already uses that count as “N lines” ([inline_label_editor.dart](/C:/code/draftable/chess-4/src/lib/widgets/inline_label_editor.dart#L163)). Fix: either change the copy to “continuation line(s)” or compute the actual number of descendant moves being relocated.

5. **Minor — Step 1 / Step 3:** `findReparentConflicts` is planned as a repository API even though the controller already has all required data in `RepertoireTreeCache` via `getChildren()`/`isLeaf()` ([repertoire.dart](/C:/code/draftable/chess-4/src/lib/models/repertoire.dart#L124), [repertoire.dart](/C:/code/draftable/chess-4/src/lib/models/repertoire.dart#L129)). The plan itself notes this in Risks, so the current step set is more complex than necessary. Fix: do the pre-check in-memory in the controller/cache, keep the DB unique index as the hard safety net, and reserve the repository addition for the atomic reroute transaction only.
