- **Verdict** — `Needs Fixes`
- **Issues**
  1. **Major — Embedded design / semantic coupling:** The new session flag is documented and named as “after a successful confirm”, but it is also set on the label-only path in [add_line_controller.dart](C:/code/draftable/chess-3/src/lib/controllers/add_line_controller.dart#L716) and then directly drives the “New Line” UI in [add_line_screen.dart](C:/code/draftable/chess-3/src/lib/screens/add_line_screen.dart#L944). That means a metadata-only save on an existing line will surface a button whose intent is “start a new line after confirming one,” even though no new line or extension was created. The design now couples two different concepts: “any confirm succeeded” and “a post-save line-entry reset is available.” That weakens naming, makes the workflow harder to infer from the code alone, and risks user-visible state that does not match the button label. Fix by either restricting `_hasConfirmedSinceLastReset` to move-persisting paths only ([add_line_controller.dart](C:/code/draftable/chess-3/src/lib/controllers/add_line_controller.dart#L766)) or renaming/reframing the feature so the controller state and UI copy describe the broader behavior explicitly. Add a controller and widget test for the label-only confirm case either way.
  2. **Minor — Hidden layout assumption / Open for extension:** The action bar is still a fixed `Row` of text-bearing actions in [add_line_screen.dart](C:/code/draftable/chess-3/src/lib/screens/add_line_screen.dart#L904), and this change adds a fifth action with no overflow strategy. The plan already identified narrow-screen risk, but the implementation leaves the row brittle: extending the workflow now requires modifying the same packed layout and hoping it still fits. That is not very open for extension, and it creates a likely overflow path on small devices. Fix by using `Wrap`, `OverflowBar`, or a horizontally scrollable action container, or by moving the post-confirm action into the app bar/menu.
  3. **Minor — Single Responsibility / file size smell:** The change continues to grow already oversized files: [add_line_controller.dart](C:/code/draftable/chess-3/src/lib/controllers/add_line_controller.dart) is 1133 lines, [add_line_screen.dart](C:/code/draftable/chess-3/src/lib/screens/add_line_screen.dart) is 870 lines, [add_line_controller_test.dart](C:/code/draftable/chess-3/src/test/controllers/add_line_controller_test.dart) is 3704 lines, and [add_line_screen_test.dart](C:/code/draftable/chess-3/src/test/screens/add_line_screen_test.dart) is 2949 lines. The new reset workflow is small, but it is being added into classes that already handle many unrelated concerns: loading, move entry, label editing, parity, reroute, undo, snackbars, and responsive layout. That makes the architecture harder to understand from code alone and raises the cost of future changes. Fix by extracting at least the post-confirm/reset policy from the controller and splitting screen tests by feature area so new workflow changes do not keep accumulating in monolithic files.
