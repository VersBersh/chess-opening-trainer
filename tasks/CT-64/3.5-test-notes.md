# CT-64: Test Notes

## Test files modified

### `src/test/controllers/add_line_controller_test.dart`
New group `'Reset for new line'` added at the end of the file (after the existing `'CT-59: isExistingLine with pending label changes'` group).

| Test | What it covers |
|------|---------------|
| `canResetForNewLine is false on initial load` | Verifies the flag starts as false on a fresh controller (AC: button not available before confirm). |
| `canResetForNewLine is false when merely following an existing line` | Distinguishes `canResetForNewLine` from `isExistingLine` -- following saved moves does NOT enable reset. |
| `canResetForNewLine is false when loading with startingMoveId` | Same distinction when the controller starts at a mid-tree position. |
| `canResetForNewLine is false before confirm and true after` | Core state transition: flag is false while buffering moves, flips to true after `confirmAndPersist()` succeeds (AC: button available after confirm). |
| `resetForNewLine clears pills and returns to starting position` | After confirm + reset: pills empty, `hasNewMoves` false, `currentFen` back to `kInitialFEN`, `canResetForNewLine` cleared (AC: clears board position, pills, and pending labels). |
| `resetForNewLine increments undoGeneration` | Ensures stale undo snackbar actions are invalidated after reset. |
| `resetForNewLine preserves board orientation` | Verifies board orientation (black) survives the reset (AC: repertoire and color selection preserved). |
| `canResetForNewLine is cleared by undo` | Calling `undoNewLine()` (which calls `loadData()`) clears the post-confirm flag. |
| `resetForNewLine returns to startingMoveId position when set` | When the controller was opened with a `startingMoveId`, reset returns to that position (not the absolute root), preserving context (AC: same context preserved). |
