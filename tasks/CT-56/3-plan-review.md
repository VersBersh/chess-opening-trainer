**Verdict** — `Needs Revision`

**Issues**
1. Critical — **Steps 2 and 4:** the proposed `LineEntryEngine.findTranspositions(resultingFen)` API cannot produce correct results after pill navigation, even though the plan explicitly relies on it in `onPillTapped()`. The engine’s notion of “current path” is the full `existingPath + followedMoves + bufferedMoves` it has accumulated, but pill taps do **not** mutate the engine; they only change `focusedPillIndex`, `currentFen`, and `preMoveFen` in the controller state. See [src/lib/controllers/add_line_controller.dart](C:/code/draftable/chess-1/src/lib/controllers/add_line_controller.dart):455 and [src/lib/services/line_entry_engine.dart](C:/code/draftable/chess-1/src/lib/services/line_entry_engine.dart):93. If the user taps back from `A-B-C-D` to `B`, the engine still thinks the active path is `A-B-C-D`, so the same-path filter and label classification will be wrong for position `B`. Suggested fix: make the transposition finder accept an explicit active-path snapshot from the controller (saved move IDs, effective labels, and current focus depth), or move the “current path” derivation into `AddLineController` instead of inferring it from engine internals.

2. Major — **Steps 2, 3, and 4:** the plan’s same-opening/cross-opening classification ignores the controller’s effective-label model and does not account for all `AddLineState` rebuild sites. In this screen, labels shown to the user come from `_pendingLabels` overlays plus buffered labels, and `aggregateDisplayName` is computed from those effective labels, not just persisted tree labels. See [src/lib/controllers/add_line_controller.dart](C:/code/draftable/chess-1/src/lib/controllers/add_line_controller.dart):203, [src/lib/controllers/add_line_controller.dart](C:/code/draftable/chess-1/src/lib/controllers/add_line_controller.dart):288, and [src/lib/controllers/add_line_controller.dart](C:/code/draftable/chess-1/src/lib/controllers/add_line_controller.dart):733. The plan says classification should read labels from `_existingPath` and `_followedMoves` only, which will become stale as soon as the user edits a saved label or adds a buffered label. Separately, once `transpositionMatches` is added to `AddLineState`, the plan only updates move/take-back/pill/load paths, but this controller rebuilds state in other places too (`updateLabel`, `updateBufferedLabel`, `flipBoard`, `flipAndConfirm`). See [src/lib/controllers/add_line_controller.dart](C:/code/draftable/chess-1/src/lib/controllers/add_line_controller.dart):680 and [src/lib/controllers/add_line_controller.dart](C:/code/draftable/chess-1/src/lib/controllers/add_line_controller.dart):767. Suggested fix: define the classification from the controller’s effective pill labels, and update the plan so every `AddLineState(...)` rebuild either recomputes or preserves `transpositionMatches` intentionally.
