## Architecture

The move tree widget is a stateless, pure-rendering component in a unidirectional data flow:

```
Controller (state + callbacks)
  -> BrowserContent (layout)
    -> MoveTreeWidget (tree flattening + list rendering)
      -> _MoveTreeNodeTile (per-row rendering)
```

**Data flow:** The controller holds `RepertoireTreeCache` (an in-memory indexed view of all moves), `expandedNodeIds` (which subtrees are open), `selectedMoveId`, and `dueCountByMoveId`. These are passed through `BrowserContent` into `MoveTreeWidget`. The widget calls the top-level `buildVisibleNodes()` to produce a flat `List<VisibleNode>`, then renders each node as a `_MoveTreeNodeTile`.

**Key invariant:** `buildVisibleNodes()` is a pure function extracted at the top level for direct unit testing without widget infrastructure. It takes `RepertoireTreeCache` and `Set<int> expanded` and returns `List<VisibleNode>`. All chain-collapsing logic goes here.

**Expand state compatibility:** The controller's `_computeInitialExpandState()` expands all unlabeled interior nodes. After chain collapsing, the chain tail is the last single-child unlabeled node before a branch or leaf. Since the tail is an unlabeled interior node (it has children -- the branch point or next nodes), it is already in the expanded set. This means expand/collapse works correctly without controller changes: the tail's ID is used for `expanded.contains()` checks, and the tail is already expanded by default.

**Notation format:** `getMoveNotation()` in `RepertoireTreeCache` computes notation from plyCount: odd ply = `"N. san"` (white), even ply = `"N...san"` (black), where `N = (plyCount + 1) ~/ 2`. The new `buildChainNotation()` uses this same formula but builds compact multi-move notation, omitting the move number prefix for black moves that immediately follow a white move in the same chain.

**Constraints:**
- No changes to `RepertoireTreeCache`, `RepertoireMove`, the controller, or `browser_content.dart`.
- The `VisibleNode` class is internal to `move_tree_widget.dart` -- changing it from `move` to `moves` only affects this file and its test file.
- `_MoveTreeNodeTile` receives pre-computed strings and callbacks, so its internal structure needs only minor updates (label reference and `hasLabel` check).
