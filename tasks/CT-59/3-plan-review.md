**Verdict** — `Needs Revision`

**Issues**
1. **Critical — Step 4:** The proposed label-only path calls `_loadData()` after saving labels. In the current controller, `_loadData()` without `leafMoveId` resets the screen to the original `startingMoveId`, and even `_loadData(leafMoveId: ...)` re-focuses the last pill. That breaks the existing “pills/board stay where the user is” behavior for label-only saves. A user who edits a label on a followed line would be jumped back to the start or to the leaf. Fix: capture the current visible line/focused move before persisting, reload against the current leaf, then restore the focused pill/board position, or update the cache/state in place instead of doing a blind reload.

2. **Major — Steps 5, 6, 7:** `ConfirmLabelsOnly` adds a new `ConfirmResult` branch, but the plan only updates `_onConfirmLine`. `_onFlipAndConfirm()` still handles only `ConfirmSuccess`, `ConfirmParityMismatch`, and `ConfirmError`, so the new result would be ignored there. More importantly, this extra result type is unnecessary complexity: the existing `_handleConfirmSuccess()` already skips undo snackbars when `insertedMoveIds` is empty. Fix: avoid `ConfirmLabelsOnly` and return a normal `ConfirmSuccess` for label-only saves, with no inserted IDs.

3. **Major — Steps 4 and 6:** The plan forgets to invalidate the existing undo snackbar path for label-only confirms. Today both `confirmAndPersist()` and `flipAndConfirm()` do `_undoGeneration++` before any successful confirm flow. Without that, a stale undo snackbar from an earlier line save remains valid after a label-only save and can roll back old moves unexpectedly. Fix: increment `_undoGeneration` before label-only persistence as well.

4. **Major — Step 3:** `persistLabelsOnly()` is proposed as a loop over `RepertoireRepository.updateMoveLabel()`. That does not match the current architecture, where multi-write operations live in repository-level transaction methods (`extendLineWithLabelUpdates`, `saveBranchWithLabelUpdates`, `rerouteLine`). It also allows partial success if one update fails midway. Fix: add a batch label-update repository method and implement the label-only path transactionally in `LocalRepertoireRepository`.

5. **Minor — Steps 11 and 12:** The test plan is too narrow for the behavior being changed. It covers the happy-path confirm, but not the highest-risk regressions introduced by this change: preserving focus/board position after label-only save, discard dialog activation when only pending labels exist, and stale undo invalidation. Fix: add tests for those cases, and only keep a `flipAndConfirm` label-only test if that code path still exists after simplifying the result handling.
