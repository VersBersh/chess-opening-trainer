# CT-65: Test Notes

## Test files modified

- `src/test/widgets/move_tree_widget_test.dart`

## Tests added

### Unit tests: `group('computeTreeIndent', ...)`

| Test | What it covers |
|------|----------------|
| depth 0 returns base padding | Verifies `computeTreeIndent(0)` returns `kTreeRowPaddingLeft` (8.0) |
| depth 1 returns base plus one level | Verifies `computeTreeIndent(1)` returns 28.0 (`kTreeRowPaddingLeft + kIndentPerLevel`) |
| depth at max (5) returns base plus 5 levels | Verifies `computeTreeIndent(5)` returns 108.0 (the cap value) |
| depth beyond max is capped | Verifies depths 6 and 10 both return 108.0, same as depth 5 |
| negative depth clamps to 0 | Verifies `computeTreeIndent(-1)` returns 8.0 (same as depth 0) |

### Widget test: inside `group('MoveTreeWidget', ...)`

| Test | What it covers |
|------|----------------|
| indentation is capped for deeply nested nodes | Builds a tree with branches at every level (depth 0 through 7+), expands all branch nodes, renders inside a `SizedBox(width: 360, height: 600)`, then inspects `Padding` widgets to assert: (1) depth-1 rows have left padding 28.0, (2) the maximum left padding is 108.0 (not 148.0 for uncapped depth 7), and (3) no row exceeds the 108.0 cap |

## Acceptance criteria coverage

| Criterion | Covered | Notes |
|-----------|---------|-------|
| Indentation stops growing beyond a configured max depth | Yes | Unit tests verify `computeTreeIndent` caps at `kMaxIndentDepth`; widget test verifies actual rendered padding is capped |
| Deep lines remain readable on 360dp-wide screens | Partially | Widget test uses `SizedBox(width: 360)` and verifies padding does not exceed 108dp, but does not measure actual text truncation or overflow. Visual readability is best verified manually. |
| Existing shallow trees are visually unchanged | Yes | Unit tests confirm depths 0-5 produce the same values as the original formula; widget test asserts depth-1 padding is 28.0 |
| Widget test verifying indentation cap behaviour | Yes | Covered by the "indentation is capped for deeply nested nodes" test |

## Criteria that could not be tested automatically

- **Visual readability on 360dp screens**: The widget test verifies that padding values are capped, but cannot assess whether the remaining text area is subjectively "readable". This requires manual visual inspection on a 360dp device or emulator.
- **Visual differentiation of deep nodes at the same indent level**: Nodes at depth 5, 6, and 7 all render at the same indent. Whether users can still understand the hierarchy is a UX concern that requires manual testing.
